# Описание ПО калькулятор ФЭМ ReFlex

## Назначение

`ReFlex Калькулятор ФЭМ` — интерактивное приложение на Streamlit для финансово-экономического моделирования ранних этапов запуска ReFlex в трех бизнес-моделях:

- `model_a` (B2B: клиника покупает парк устройств);
- `model_b` (B2B2C: пациент арендует, клиника получает комиссию);
- `model_ab` (гибрид A+B).

Цель калькулятора — помочь команде быстро проверять unit economics, динамику cash flow, целевую безубыточность и чувствительность параметров.

## Архитектура

Основные слои:

- `app.py` — UI, orchestration, state management, вызовы расчетов и графиков;
- `models/` — расчетное ядро;
- `visualization/` — построение графиков и KPI-блоков;
- `utils/` — валидация, экспорты, snapshot-конфигурации;
- `config/` — дефолтные параметры и сценарии;
- `tests/` — unit-тесты и проверка консистентности формул.

## Поток работы приложения

1. Загрузка `defaults.json` и `scenarios.json`.
2. Инициализация `st.session_state` для:
   - `saved_params` по моделям,
   - `custom_fixed_costs`, `custom_variable_costs`, `custom_revenue`,
   - статусов stage-gate (`A01..A04`),
   - `patient_mode_{model}` — режим потока пациентов (`auto` / `manual`),
   - `manual_patients_{model}` — ручные данные по когортам (список длиной до 36).
3. Ввод параметров в sidebar:
   - выбор модели/сценария/горизонта;
   - revenue/fixed/variable/assumptions;
   - режим потока пациентов и кнопка таблицы клиентской базы;
   - кастомные статьи затрат;
   - импорт/экспорт snapshot.
4. Автосохранение параметров текущей модели в `saved_params[model_type]`.
5. Отображение модального окна клиентской базы (если нажата кнопка).
6. Выполнение расчетов в `models/*` (с подстановкой ручных данных, если режим `manual`).
7. Отображение KPI, графиков, таблиц и блока интерпретации допущений.
8. Экспорт результатов (Excel/JSON/Markdown).

## Логика расчетов

### Режим потока пациентов (ручной / автоматический)

Для каждой бизнес-модели доступен переключатель **«Режим потока пациентов»**:

- **Автоматический** — количество новых пациентов рассчитывается по формуле роста:
  ```
  new_patients[1] = patients_per_clinic_month1 * num_clinics
  new_patients[m] = new_patients[m-1] * (1 + growth_rate)  при m > 1
  ```
- **Ручной ввод** — пользователь задаёт количество новых пациентов на клинику отдельно для каждого месяца. Данные хранятся в `st.session_state['manual_patients_{model}']` как список длиной до 36 (по числу месяцев горизонта). При передаче в движок расчётов умножаются на число клиник в пачке.

В **обоих режимах** параметр `rehab_duration_months` определяет ширину когортного окна: активная когорта = сумма новых пациентов за последние `D` месяцев.

**Таблица клиентской базы** (открывается кнопкой «Ручной ввод / Посмотреть клиентскую базу»):
- Строки = когорты (когорта i = пациенты, стартующие в месяц i).
- Столбцы = месяцы M1..Mn (длина горизонта = `num_months`).
- Цветной блок когорты = `rehab_duration_months` ячеек с момента старта.
- Верхняя строка = суммарные активные пациенты на клинику по месяцам.
- В автоматическом режиме таблица только для просмотра.
- Ручной ввод применяется ко всем клиникам одинаково; для пачек с более поздним стартом используется относительный месяц когорты.

### Model A (`models/revenue.py`)

- Выручка месяца 1: `setup_fee * num_clinics * devices_per_clinic` + подписка.
- В последующих месяцах setup не начисляется, остается подписка.
- Количество устройств: `num_clinics * devices_per_clinic`.
- Если задан поток пациентов (авто или ручной), модель отслеживает когорты пачек независимо и расширяет парк при дефиците.

### Model B (`models/revenue.py`)

- Считается поток **новых пациентов** по месяцам (месяц 1 — базовый, далее рост `growth_rate`).
- **Активные пациенты** формируются когортно как сумма последних `rehab_duration_months` когорт.
- Выручка:
  - `gross_revenue = active_patients * rental_price`
  - `clinic_commission = gross_revenue * clinic_commission_rate`
  - `net_revenue = gross_revenue - clinic_commission`
- Пул устройств расширяется при дефиците: `additional_devices = max(0, active - devices_in_pool_prev)`.

### Model A+B (`models/revenue.py`)

- В месяц 1 учитывается продажа стартового парка.
- Далее к модели A добавляется rental-компонент с логикой когорт из модели B.
- Подписка начисляется на фактический пул устройств в эксплуатации.
- `total_revenue = setup + subscription + rental_net`.

### Затраты (`models/costs.py`)

- `fixed_costs`: зарплаты, инфраструктура, офис, юридические, прочие + кастомные fixed.
- `variable_costs`: COGS, логистика, поддержка, CAC + кастомные variable.
- Кастомные статьи учитываются в итоговом `total_costs`.

### Cash Flow (`models/cash_flow.py`)

- `cash_flow = revenue - total_costs`
- `cumulative_cash_flow` считается накопительно по месяцам.

### Breakeven и Target Pricing (`app.py`)

- Breakeven определяется по моменту перехода `cumulative_cash_flow >= 0`.
- Для target pricing используется линейная аппроксимация:
  - `Revenue_required = TotalCosts(T) * (1 + target_margin)`
  - `Revenue(T) = BaseRevenue + Coeff * Price`
  - `Price_min = (Revenue_required - BaseRevenue) / Coeff`
- Для `model_ab` поддержаны режимы:
  - `rental_only`,
  - `subscription_only`,
  - `both_scaled`.

### Unit Economics (`models/unit_economics.py`)

Для моделей с пациентской монетизацией рассчитываются ключевые метрики (`LTV`, `CAC`, `LTV/CAC`, `Payback`).

### Sensitivity (`models/sensitivity.py`)

Считается изменение итогового cash flow при варьировании параметров на заданный процент.

## Допущения и stage-gate

- Stage-gate `A01..A04` ведется через статус `hypothesis/in_validation/validated`.
- В UI есть mini what-if блок, который показывает численную дельту ключевых метрик для допущений `A01..A04`.
- Эти дельты служат для интерпретации гипотез, а не заменяют полноценную валидацию на интервью/пилотах.

## Импорт/экспорт конфигурации

Реализован snapshot-цикл (`utils/config_snapshot.py`):

- `build_config_snapshot` — формирует переносимый JSON-снимок всех настроек.
- `preflight_config_snapshot` — проверяет совместимость схемы, недостающие/лишние поля.
- `apply_config_snapshot` — применяет нормализованный payload в `session_state`.

Поддерживаются:

- выгрузка snapshot JSON;
- загрузка snapshot JSON;
- обработка неизвестных и отсутствующих полей с предупреждениями.

## Визуализация и графики

В `visualization/charts.py` доступны:

- `График 1`: динамика cash flow;
- `График 2`: структура revenue за **выбранный месяц**;
- `График 3`: структура costs;
- `График 4`: breakeven;
- `График 5`: sensitivity;
- `График 6`: unit economics;
- отдельный сворачиваемый график когортной динамики (новые/активные/вышедшие пациенты).

## Проверка корректности

Рекомендуемый минимальный контур:

```bash
cd "reflex-calculator"
./venv/bin/python -m pytest tests -q
./venv/bin/python -m py_compile app.py
```

Тесты покрывают:

- расчет выручки по моделям (`tests/test_revenue.py`);
- консистентность формульного трейса (`tests/test_formula_trace.py`);
- snapshot-импорт/экспорт (`tests/test_snapshot_config.py`);
- форматирование (`tests/test_formatters.py`);
- графики выбора месяца и когорт (`tests/test_charts.py`).

## Ограничения текущей версии

- Горизонт моделирования ограничен 36 месяцами в UI.
- Часть интерпретационных блоков (what-if по допущениям) работает как аналитический слой поверх базовой модели.
- Для внешней коммуникации результаты по A01–A04 должны рассматриваться с учетом статуса валидации.
- Ручные данные клиентской базы не включаются в snapshot-конфигурацию (сохраняются только в рамках текущей сессии).
- Sensitivity analysis в ручном режиме потока варьирует только ценовые и затратные параметры; параметры роста (`patients_per_clinic_month1`, `growth_rate`) заморожены и не участвуют в варьировании.
